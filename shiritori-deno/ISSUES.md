# 実装上の未決事項・既知の制限

実装手順書の「後で詰める」に対応する、現時点での決定内容と残課題。

## 1. 判定の優先順位（決定済み）

`server.ts` の `handlePostShiritori` は以下の順で判定する:

1. 空入力チェック
2. 接続判定（前の単語の末尾 = 今回の単語の先頭）
3. 実在チェック（Wikipedia OpenSearch API）
4. 既出チェック → ゲーム終了
5. 「ん」チェック → ゲーム終了
6. 通過 → 履歴に追加

この順序により、「既出かつ『ん』で終わる単語」を入力した場合は
**既出（DUPLICATE）が優先**される。実在しない単語は既出/「ん」判定より
先に弾かれるため、「実在しないが既出でもある」ケースは既出判定に到達しない。

## 2. 長音「ー」の扱い（未解決・既知の制限）

しりとりの慣例では、単語末尾が長音「ー」の場合、直前の母音（例:
「タクシー」→ 'い' 音）を使って接続判定するのが一般的だが、この実装では
**「ー」をそのまま1文字として比較している**（正規化していない）。
そのため「タクシー」の次に「い」から始まる単語を入力しても接続エラーになる。

対応する場合は、単語の読み（ひらがな変換）が必要になり、Wikipedia API
だけでは読みを取得できないため別途実装が必要。現時点では対応を見送っている。

## 3. 小書き文字の扱い（部分対応）

「っ/ゃ/ゅ/ょ」などの小書き文字は、**接続判定の境界文字のみ**大文字相当に
正規化して比較している（`normalizeEdgeChar`）。単語自体の表記は変更せず、
履歴・既出判定ではそのままの文字列で比較する。

## 4. Wikipedia API 失敗時のフォールバック方針（決定済み）

`wordExists()` がネットワークエラー・タイムアウト・非 200 応答の場合は
`null` を返し、呼び出し側はこれを **fail-open（実在チェックをスキップして
通過させる）** として扱う。API の不安定さでゲームが進行不能になることを
避けるため。ただし、これにより「本来実在しない単語」が通ってしまう可能性は
残る。

## 5. ゲーム終了後の再入力

`isGameOver` が true の状態で `POST /shiritori` を呼ぶと `GAME_OVER`
エラーで拒否する。再開には `POST /reset` が必要（フロントの「リセット」
ボタン、または「モード選択に戻る」→ 再度スタートで対応）。

## 6. モードは現状 1 種類のみ

`public/app.js` の `MODES` 配列は通常モードのみを定義しているが、
`{ id, name, description }` の形で他モードを追加できる構造にしてある。
モードごとにルール設定（例: 制限時間、NG 単語リストなど）を持たせる場合は
サーバー側にも `mode` をパラメータとして渡す仕組みが別途必要になる
（現状サーバーはモードを区別していない）。
