# しりとりアプリ

Deno + TypeScript で実装した、ブラウザで遊べるしりとりゲームです。
モードを選んでから遊ぶ構成になっており、「通常モード」「CPU対戦」「タイムアタック」「食べ物縛り」
「しりとり×暗記」を実装しています。

アプリ本体は [`shiritori-deno/`](./shiritori-deno) にあります。

## デプロイ先URL

https://jigjp-intern-shiritori-hthfajd7c1d7.rk200781.deno.net/

ローカルでは `shiritori-deno/` ディレクトリで
`deno run -A --watch server.ts` を実行すると `http://localhost:8000` で動作します。

## 実装した機能

### 画面構成
1. モード選択画面 — モードを配列で定義し、そこから自動でボタンを生成している
   （`shiritori-deno/public/app.js` の `MODES`）。今後モードを追加する場合は
   配列に要素を追加するだけでよい構造にしてある。
2. モード説明画面 — 選んだモードの説明文と「ゲームスタート」ボタンを表示
3. ゲーム画面 — しりとり本体

### デザイン
CSSカスタムプロパティで色・余白・角丸・影をトークン化し、ページ全体に一貫した
見た目を持たせている（`shiritori-deno/public/styles.css`）。

- グラデーション背景の上に、中央寄せの角丸カード（`.card`）でコンテンツを配置
- モード選択画面は、各モードに絵文字アイコン（🔤 通常／🤖 CPU対戦／⏱️ タイムアタック／
  🍙 食べ物縛り／🧠 しりとり×暗記）を付けたカードのリストにし、ホバーで浮き上がる
  アニメーションを付けている
- 直前の単語・CPUの単語はピル型のチップ（`.current-word-chip`）で表示し、
  スコア・ターン数・単語数はバッジ風のチップ（`.stat-chip`）で表示
- 単語履歴は番号付きリストではなく「→」でつながるチップの連鎖として表示し、
  しりとり＝単語の鎖というテーマを見た目にも反映
- 勝敗・終了バナーは結果に応じて色分け（CPU対戦の勝利・タイムアタックの3分
  完走は緑の成功色、それ以外のミスによる終了は赤の失敗色）
- エラーメッセージには警告アイコンを付け、ボタン・入力欄はホバー/フォーカス/
  disabled状態を整理し、押下時に軽く縮むアニメーションを付けている
- 420px以下の狭い画面向けにレイアウトを調整するレスポンシブ対応を入れている

### 必須仕様
- 直前の単語の表示
- 単語の入力・送信フォーム
- 接続判定（前の単語の末尾と入力単語の先頭が一致するか）とエラー表示
- 「ん」で終わる単語を入力するとゲーム終了
- すでに使った単語を入力するとゲーム終了
- リセット機能（ゲーム中・終了後どちらでも実行可能）

### 追加機能
- 単語履歴の一覧表示
- Wikipedia OpenSearch API を使い、実在しない単語を弾く
  （API が失敗した場合はゲーム進行を止めないよう、実在チェックをスキップして
  通過させる fail-open 方式にしている）
- kuromoji（形態素解析ライブラリ）で入力単語の読みを取得し、
  読みベースで接続判定を行う（長音「ー」・カタカナ表記・漢字表記に対応）。
  辞書をバンドルしてサーバー内で完結させているため外部APIには依存しない
  （詳細は [`shiritori-deno/README.md`](./shiritori-deno/README.md) を参照）。
  食べ物縛りモードの実装時に、「じゅーす」「ぱーてぃー」のように小書きの
  ゃゅょ/ぁぃぅぇぉの直後に長音「ー」が続く語で母音解決が漏れるバグ
  （`KANA_VOWEL` に小書き文字のエントリが無かったため）を発見し、修正した

判定ロジックの詳細・既知の制限は GitHub Issues で管理している。

### CPU対戦
辞書相手にどれだけ続けられるかを競う、CPU対戦のしりとりモード。

- CPU が `shiritori-deno/data/words.json`単語リストから
  単語を出題し、プレイヤーは制限時間10秒以内に続く単語を入力して応答する
- 判定順序: 接続判定 → 実在チェック → 既出チェック（負け） → 「ん」判定（負け）
- タイムアウト・既出単語・「ん」で終わる単語のいずれかでゲーム終了（プレイヤーの負け）
- CPU側の候補が尽きた場合はプレイヤーの勝利（`data/words.json` と `server.ts` の `pickCpuWord` を読むと、CPUに勝てるかも）
- 応答速度に応じたスコア加算（基本点 + 残り時間ボーナス）とターン数を記録し、
  ゲーム終了時に結果を表示する
- CPU の単語選択で、濁点・半濁点付きの頭文字（が・ざ・だ・ば・ぱ行など）の候補が
  尽きた場合は清音側（が→か、づ→つ、など）にフォールバックする
  （「ぢ」「づ」は現代仮名遣い上、連濁でしか現れず単語の先頭に立たないため
  実在する単語が存在せず、必ずこのフォールバックが働く）
- 実在チェックは、自前の単語リストに加えて Jisho（JMdict辞書API）と
  Wikipedia OpenSearch API を組み合わせて判定する。Wikipedia
  だけだとひらがな表記の一般名詞（ひらがなタイトルの記事・リダイレクトが
  存在しないもの）を実在しないと誤判定することがあったため、
  いずれか一方が実在を確認すれば通過、両方が「存在しない」と
  判定した場合のみ却下する方式にしている
- CPU の返答には 1〜2秒のランダムな演出ディレイを入れており、
  この演出はフロントエンド側だけで完結させている（サーバー側の応答自体は遅延させない）

### タイムアタック
CPUなしで1人プレイする、3分間ひたすら単語をつなげ続けるモード。

- ラウンド開始時に CPU対戦 と共用の `data/words.json` から単語を1つランダムに選び、
  最初の単語として提示する（単語リストが読み込めない場合は最初の単語なしで、
  プレイヤーが任意の単語から始められる）
- 接続判定・1文字禁止・実在チェックは通常モード／CPU対戦と共通の
  `validateWordReading` / `checkWordExists` をそのまま利用する
- 制限時間3分が経過したら**正常終了**として、そこまでのスコア・単語数を表示する
  （CPU対戦のタイムアウトとは異なり、ミス扱いにはしない）
- 「ん」で終わる単語・すでに使った単語を入力した場合はその時点でゲーム終了（ミス扱い）
- 実在しない単語・接続していない単語・1文字の単語はエラー表示のみで、
  ゲームを終了させずに再入力できる
- 応答速度に応じたスコア加算（CPU対戦と同じ計算式）と単語数を記録する
- 残り30秒以下でタイマー表示が警告色に変わる

### 食べ物縛り
通常モードに「食べ物・飲み物の単語しか使えない」縛りを加えたモード。

- カテゴリ判定は `shiritori-deno/data/foods.json`（食べ物・飲み物のひらがな単語
  リスト、479語）への完全一致で行う。表記が異なっても同じ語だと判定できるよう、
  入力そのものだけでなく正規化した読みでもリストと照合する（例:「寿司」「スシ」
  のどちらで入力しても「すし」としてリストと一致する）
- リストにない単語（カテゴリ外）を入力した場合はエラー表示のみで、ゲームは
  終了せず再入力できる
- それ以外のルール（接続判定・1文字禁止・実在チェック・「ん」で終了・既出で
  終了・リセット機能）は通常モードと共通の `validateWordReading` /
  `checkWordExists` をそのまま利用し、状態も通常モードとは完全に分離している
- `data/foods.json` は日本・中華・韓国・イタリア・フランス・アメリカ・スペイン・
  インド・タイ・ベトナム・中東・メキシコ・ドイツ・北欧・東南アジア・南米料理や、
  果物・野菜・きのこ・肉・魚介・パン・穀類・お菓子・デザート・飲み物まで幅広い
  ジャンルの単語を独自に手作業で収録したオリジナルリストで、外部データセットからの
  転載ではない（ライセンス表記の必要なし）

### しりとり×暗記
直前の単語も履歴も画面に一切表示せず、プレイヤー自身の記憶力を試すモード。

- プレイヤーは毎ターン、これまでにつないだ単語を**頭から全部スペース区切りで
  入力し直し**、末尾に新しい単語を1つ追加する（例: 1週目「りんご」→2週目
  「りんご ごりら」→3週目「りんご ごりら らっぱ」）
- サーバー側は入力された単語数が「これまでの履歴数＋1」と一致するか、かつ
  末尾を除く部分がこれまでの履歴と単語ごとに完全一致するかを検証する。
  記憶違い（数が合わない・順番や表記が違う）はその時点で即ゲーム終了とする
  （「ん」・既出と同じ扱い。暗記モードなので中途半端な間違いを見逃さない）
- 新しく追加された最後の1単語には、通常モードと共通の `validateWordReading`
  （接続判定・1文字禁止）と `checkWordExists`（実在チェック）をそのまま適用し、
  こちらは他モードと同じく続行可能なエラー表示にとどめる
- 「ん」で終わる単語・すでに使った単語を追加した場合もその時点でゲーム終了
- プレイ中はサーバーのレスポンスにも単語履歴を含めない（`wordCount`という
  件数だけを「次は◯単語目」のヒントとして返す）。ゲーム終了時のみレスポンスに
  全履歴を含めて画面に表示する
- 状態(`memoryState`)は他モードと完全に分離している

## トラブルシューティング事例：本番環境だけで接続判定が誤動作する問題

ローカルでは問題なく動くのに、本番デプロイ後だけ「しりとりが繋がっているのに
繋がっていないと言われる」「実在する単語なのに実在しないと言われる」という
不具合報告があり、以下の流れで原因調査・対応を行った。

1. **症状の再現**: 本番URLに対して `curl` で実際のリクエストを送り、
   `りんご→ゴリラ`（カタカナ⇄ひらがな接続）や `新幹線`（漢字表記で読みが「ん」
   終わり）など、読みの正規化が絡む単語で再現することを確認。ローカル環境
   （`.env` に `YAHOO_APP_ID` 設定済み）では同じ入力が正常に通ることも確認し、
   「ローカルと本番で挙動が違う」こと自体を切り分けの起点にした。
2. **実在チェックの不具合はコードのデプロイ遅れと判明**: 調べた結果、直近の
   修正コミットがまだ本番に反映されていなかっただけと判明。GitHubに
   webhook／deploymentsが登録されていないことも確認し、push だけでは
   自動デプロイされない構成であることを特定した。
3. **接続判定の不具合は再デプロイ後も再現**: コードを再デプロイし、環境変数
   `YAHOO_APP_ID` の設定（Secret種別・Contexts: All）も確認したが、
   それでも本番だけ読み取得が失敗する状態が続いた。
4. **原因特定のため一時的にログを追加**: `getReading` 内の失敗経路
   （非OKレスポンス・例外・APIエラー）に理由を出力するログを追加してデプロイし、
   本番ログを確認してもらったところ、Yahoo! APIが403 Forbiddenで
   「欧州経済領域（EEA）およびイギリスからはご利用いただけません」という
   案内ページを返していることが判明した。
5. **根本原因**: Yahoo! JAPANは2022年4月よりEEA・イギリスからのアクセスを
   全面的に遮断しており、Deno Deployのサーバーがその地域と判定される
   ネットワーク経路でYahoo!にアクセスしていたため、API呼び出し自体が
   常にブロックされていた。これは自分たちのコードや設定では制御できない、
   デプロイ先インフラ側の制約だった。
6. **対応方針の検討**: 「ローカル形態素解析ライブラリへの切り替え」「表記
   ベース判定への後退（読みAPI自体をやめる）」「様子見」の3案を比較し、
   外部APIの地域制限という制御不能な要因に判定ロジックの正しさが左右される
   状態を避けるため、kuromoji（ローカルで動く形態素解析ライブラリ）へ
   切り替える方針を選択した。
7. **実装・検証**: kuromoji導入後、`りんご→ゴリラ`・`タクシー→いす`（長音）・
   `新幹線`（漢字の「ん」判定）・`ねこ→こおり`（実在チェック）など、
   問題が起きていた全パターンをローカルで再テストし、正常動作を確認した。

この件の詳細な経緯は [`shiritori-deno/README.md`](./shiritori-deno/README.md)
にも記載している。

## AIの活用について

このプロジェクトは、私（人間）が企画・仕様決定・意思決定を担当し、
Claude Code が実装を担当するというスタイルで進めた。「何を作るか」
「どう判断するか」「見つかった問題にどう対応するか」は私が方向性を示し、
実際のコードを書く・調査する・検証するという実務は Claude Code に任せている。

具体的には、まず画面構成（モード選択→モード説明→ゲーム画面）としりとりの
必須仕様（接続判定・「ん」判定・既出判定・リセット）を私から指示し、
`server.ts` のAPI実装や判定順序の設計、Wikipedia OpenSearch API との連携、
フロントエンドの画面切り替えJSといった実装は Claude Code に行わせた。
実装方針や優先順位に迷う点（例えば「実在チェックに失敗したときゲームを
止めない方が良いのでは」といった判断）は、私と Claude Code で相談しながら
決定し、後から見返せるように GitHub Issues として記録している。

その後、CPUと対戦して語彙力を診断する「CPU対戦」モードを追加する際も、
単語リストの生成やスコア計算・タイマー演出・CPUの単語選択ロジックといった
実装一式を Claude Code に任せ、私は「実際に対戦して見つけた不具合」（実在する
単語が却下される、など）を報告し、原因調査と修正を依頼するというやり取りを
繰り返した。

特に印象に残っているのは、本番デプロイ後にだけ「しりとりが繋がっている
はずなのに繋がっていないと言われる」という不具合が起きたときの対応である。
ローカルでは再現しないという報告をもとに Claude Code に調査を依頼したところ、
デバッグ用ログの追加・本番ログの確認という段階を踏んで、Yahoo! JAPANが
2022年からEEA（欧州経済領域）・イギリスからのアクセスを遮断しており、
デプロイ先サーバーがその地域と判定されて読み取得APIが403で弾かれていた、
という自分たちでは制御できない外部要因が根本原因だと突き止めた。
この報告を受けて、外部APIに依存しないkuromoji（ローカル形態素解析ライブラリ）
への置き換えを私から指示し、実装・動作確認までを Claude Code が行った。

最後に、実装当初に「決定事項の記録」として起票していた GitHub Issues を
棚卸しし、すでに実装で解決済みのもの・現状の設計を正式な仕様として確定して
良いものを私が判断してクローズを指示、未対応だった設計上の論点（小書き文字の
正規化範囲、モード追加の仕組み）についても対応方針を私が決め、実装を
Claude Code に依頼して仕上げている。
